# Feature Specification: SSR + CSR 하이브리드 렌더링 (TanStack Query)

**Feature Branch**: `022-ssr-csr-hybrid`
**Created**: 2026-03-05
**Status**: Draft
**Input**: User description: "SSR 환경에서 tanstack query를 사용해 SSR + CSR 하이브리드 렌더링으로 변경하고싶어."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 스킬 상세 모달 즉시 재표시 (Priority: P1)

사용자가 스킬 상세 팝업을 닫았다가 같은 스킬을 다시 클릭하면, 이전에 로드된 데이터가 즉시 표시되고 백그라운드에서 최신 데이터로 갱신된다.

**Why this priority**: 현재 커스텀 SWR 캐시(5분 TTL Map)를 표준 쿼리 라이브러리로 교체하는 것이 가장 높은 코드 품질 개선과 유지보수성 향상을 가져온다. 가장 복잡한 클라이언트 상태 관리를 정리하는 핵심 스토리.

**Independent Test**: 스킬 상세 모달을 열고 닫은 뒤 같은 스킬을 다시 클릭하여 즉시 데이터가 표시되는지 확인. 피드백 목록의 페이지네이션도 캐시된 상태로 유지되는지 검증.

**Acceptance Scenarios**:

1. **Given** 사용자가 스킬 상세 모달을 열어 데이터가 로드된 상태, **When** 모달을 닫고 5분 이내에 같은 스킬을 다시 클릭, **Then** 이전 데이터가 즉시 표시되고 백그라운드에서 최신 데이터로 갱신됨
2. **Given** 스킬 상세 모달이 열린 상태에서 피드백을 작성, **When** 피드백 제출, **Then** 작성한 피드백이 즉시 목록에 나타남 (서버 응답 전 낙관적 표시)
3. **Given** 스킬 상세 데이터가 캐시에 있는 상태, **When** 캐시 유효 시간(5분)이 경과, **Then** 다음 접근 시 서버에서 새 데이터를 가져옴

---

### User Story 2 - 대시보드 네비게이션 즉시 표시 (Priority: P2)

사용자가 대시보드에서 다른 페이지로 이동했다가 돌아오면, 이전에 로드된 스킬 목록과 북마크 상태가 즉시 표시된다. 서버에서 초기 렌더링한 데이터는 로딩 상태 없이 바로 보여진다.

**Why this priority**: 대시보드는 가장 자주 방문하는 메인 페이지. 서버에서 미리 가져온 데이터를 클라이언트 캐시로 연결하여 초기 렌더링 성능 유지 + 재방문 시 즉시 표시라는 두 가지 이점 제공.

**Independent Test**: 로그인 후 대시보드 진입 시 로딩 스피너 없이 스킬 목록이 바로 표시되는지 확인. 다른 페이지 이동 후 대시보드 재진입 시 캐시된 데이터 즉시 표시 확인.

**Acceptance Scenarios**:

1. **Given** 사용자가 처음 대시보드에 접속, **When** 페이지가 로드됨, **Then** 서버에서 미리 가져온 스킬 목록이 로딩 상태 없이 즉시 렌더링됨
2. **Given** 대시보드 데이터가 캐시에 있는 상태, **When** 다른 페이지에서 대시보드로 돌아옴, **Then** 캐시된 스킬 목록이 즉시 표시되고 백그라운드에서 최신 데이터 갱신
3. **Given** 사용자가 스킬을 북마크 토글, **When** 북마크 버튼 클릭, **Then** UI가 즉시 반영되고 실패 시 자동 롤백됨
4. **Given** 대시보드와 내 에이전트 페이지가 같은 북마크 데이터를 공유, **When** 대시보드에서 북마크 추가, **Then** 내 에이전트 페이지에서도 해당 스킬이 바로 반영됨

---

### User Story 3 - 관리자 페이지 실시간 반영 (Priority: P3)

관리자가 스킬을 추가/수정/삭제하거나 사용자 역할을 변경하면, 전체 페이지 새로고침 없이 목록이 즉시 업데이트된다.

**Why this priority**: 관리자 페이지는 트래픽이 낮지만, CRUD 작업 후 즉시 반영은 관리 워크플로우 효율성을 크게 높인다. 아키텍처 일관성 측면에서도 중요.

**Independent Test**: 관리자 페이지에서 스킬 추가 후 목록에 즉시 나타나는지 확인. 역할 변경 후 멤버 목록에 반영되는지 확인.

**Acceptance Scenarios**:

1. **Given** 관리자가 스킬 목록 페이지에서 새 스킬을 추가, **When** 저장 완료, **Then** 스킬 목록에 전체 새로고침 없이 새 스킬이 나타남
2. **Given** 관리자가 멤버의 역할을 변경, **When** 역할 변경 완료, **Then** 멤버 목록에 변경된 역할이 즉시 반영됨

---

### Edge Cases

- 네트워크가 느리거나 끊긴 상태에서 캐시된 데이터는 계속 표시되고, 재연결 시 자동으로 최신 데이터를 가져옴
- 서버 데이터와 캐시 데이터 간 불일치 시 (다른 관리자가 스킬을 삭제한 경우), 백그라운드 갱신을 통해 자연스럽게 동기화됨
- 브라우저 탭을 오래 방치했다가 돌아오면 자동으로 데이터를 새로 가져옴 (window focus refetch)
- 사용자 인증이 만료된 상태에서 쿼리 실패 시, 로그인 페이지로 적절히 리다이렉트됨

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 서버에서 미리 가져온 데이터를 클라이언트 캐시에 전달(hydration)하여 초기 렌더링 시 로딩 상태가 발생하지 않아야 함
- **FR-002**: 클라이언트 캐시는 데이터 유형별로 적절한 유효 시간을 가져야 하며, 유효 시간 경과 후 백그라운드에서 자동 갱신해야 함
- **FR-003**: 데이터 변경 작업(북마크 토글, 피드백 작성, 관리자 CRUD)은 서버 응답 전에 UI에 즉시 반영(낙관적 업데이트)되어야 함
- **FR-004**: 낙관적 업데이트가 서버에서 실패하면 자동으로 이전 상태로 롤백되어야 함
- **FR-005**: 같은 데이터를 참조하는 서로 다른 페이지(대시보드 ↔ 내 에이전트)에서 캐시가 공유되어 일관된 상태를 보여야 함
- **FR-006**: 기존 서버 컴포넌트의 데이터 가져오기 패턴(병렬 실행, 요청별 중복 제거)은 그대로 유지되어야 함
- **FR-007**: 브라우저 탭에 포커스가 돌아오면 stale 데이터를 자동으로 백그라운드 갱신해야 함
- **FR-008**: 스킬 상세 모달의 피드백 목록은 무한 스크롤/페이지네이션을 지원하며, 이전에 로드된 페이지가 캐시에 보존되어야 함
- **FR-009**: 인증/로그인 페이지는 하이브리드 렌더링 대상에서 제외하며, 기존 서버 액션 + 리다이렉트 방식을 유지해야 함

### Key Entities

- **Query Cache**: 클라이언트 메모리에 저장되는 쿼리 결과 캐시. 쿼리 키로 식별되며 유효 시간(staleTime)과 가비지 컬렉션 시간(gcTime)을 가짐
- **Hydration State**: 서버에서 프리페치한 데이터를 직렬화하여 클라이언트로 전달하는 상태 객체. 초기 렌더링 시 캐시를 미리 채움

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 대시보드 초기 로드 시 로딩 스피너가 표시되지 않음 (서버 프리페치 데이터가 즉시 렌더링)
- **SC-002**: 스킬 상세 모달 재방문(5분 이내) 시 데이터가 200ms 이내로 표시됨
- **SC-003**: 북마크 토글 시 UI가 100ms 이내로 반영되며, 실패 시 자동 롤백됨
- **SC-004**: 페이지 간 이동(대시보드 ↔ 내 에이전트) 시 캐시된 데이터가 즉시 표시됨
- **SC-005**: 기존 Playwright E2E 테스트가 모두 통과함 (기능 회귀 없음)
- **SC-006**: 번들 사이즈 증가가 20KB(gzip) 이하

## Assumptions

- TanStack Query v5를 사용하며, Next.js App Router의 `HydrationBoundary` + `dehydrate` 패턴을 따름
- 서버 컴포넌트에서의 프리페치는 요청별 QueryClient 인스턴스를 생성하여 수행
- 클라이언트 쿼리 함수(queryFn)는 기존 서버 액션을 호출하여 데이터를 가져옴
- React `cache()`를 사용한 서버 사이드 요청 중복 제거는 그대로 유지
- `revalidatePath()`는 안전망으로 유지하되, 클라이언트 캐시 무효화와 병행 운영
