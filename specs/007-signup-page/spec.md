# Feature Specification: 회원가입 페이지 디자인 리뉴얼

**Feature Branch**: `007-signup-page`
**Created**: 2026-03-04
**Status**: Draft
**Input**: User description: "회원가입 페이지 구현 — signup.html 디자인 참고, 로그인 페이지 헤더/배경 컴포넌트 재사용"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 신규 사용자 회원가입 완료 (Priority: P1)

신규 사용자가 회원가입 페이지에 접속하여 이름·이메일·비밀번호를 입력하고 계정을 생성한다. 페이지는 기존 로그인 페이지와 동일한 헤더·배경 컴포넌트를 사용하며 signup.html의 글래스카드 레이아웃을 따른다.

**Why this priority**: 회원가입 폼의 핵심 기능이자 서비스 진입점이므로 가장 높은 우선순위를 가진다.

**Independent Test**: `/signup` 경로를 방문하여 이름·이메일·비밀번호·비밀번호 확인을 입력하고 "회원가입" 버튼을 클릭하면 계정이 생성되고 성공 상태로 전환된다.

**Acceptance Scenarios**:

1. **Given** 비로그인 사용자가 `/signup` 페이지를 방문했을 때, **When** 유효한 이름·이메일·비밀번호(일치)를 입력하고 제출하면, **Then** 계정이 생성되고 사용자는 성공 피드백을 받는다.
2. **Given** 사용자가 폼을 제출했을 때, **When** 비밀번호와 비밀번호 확인 값이 다르면, **Then** 일치하지 않는다는 오류 메시지가 표시되고 제출이 중단된다.
3. **Given** 사용자가 폼을 제출했을 때, **When** 이미 존재하는 이메일로 가입을 시도하면, **Then** OTP 인증 화면 대신 이미 가입된 이메일 안내 메시지와 로그인 페이지 이동 버튼이 표시된다.

---

### User Story 2 - 페이지 레이아웃 및 시각적 일관성 확인 (Priority: P2)

회원가입 페이지가 로그인 페이지와 동일한 헤더·배경을 공유하며, signup.html의 글래스카드 UI(최대 너비 480px, 라운드 코너, 반투명 배경)를 정확히 구현한다.

**Why this priority**: 로그인·회원가입 페이지 간 시각적 일관성이 브랜드 품질에 직결되며, 컴포넌트 재사용은 코드 중복을 방지한다.

**Independent Test**: 페이지 방문 시 헤더(로고 + "ELUO AI SKILL HUB" + 버튼), BackgroundBeamsWithCollision 배경, 글래스카드 폼이 모두 정상 렌더링된다.

**Acceptance Scenarios**:

1. **Given** 사용자가 `/signup` 페이지를 방문하면, **When** 페이지가 로드되면, **Then** 로그인 페이지와 동일한 헤더 레이아웃(로고, 서비스명, 버튼)이 표시된다.
2. **Given** 사용자가 `/signup` 페이지를 방문하면, **When** 페이지가 로드되면, **Then** 로그인 페이지와 동일한 BackgroundBeamsWithCollision 배경 컴포넌트가 표시된다.
3. **Given** 데스크탑 환경에서 페이지를 방문하면, **When** 폼 카드가 렌더링되면, **Then** 최대 너비 480px의 글래스카드 스타일(반투명, 블러, 라운드 코너)로 표시된다.

---

### User Story 3 - 로그인 페이지 이동 링크 (Priority: P3)

이미 계정이 있는 사용자가 회원가입 페이지 하단의 "로그인" 링크를 클릭하여 `/signin` 페이지로 이동할 수 있다.

**Why this priority**: 사용자 내비게이션 편의 기능으로, P1·P2 완료 후 구현해도 MVP가 성립한다.

**Independent Test**: "이미 계정이 있으신가요? 로그인" 링크를 클릭하면 `/signin` 페이지로 이동한다.

**Acceptance Scenarios**:

1. **Given** 사용자가 `/signup` 페이지에 있을 때, **When** "로그인" 링크를 클릭하면, **Then** `/signin` 페이지로 이동한다.

---

### Edge Cases

- 비밀번호 필드가 비어 있을 때 제출하면 어떻게 되는가? → 필수 입력 오류 메시지 표시
- 이메일 형식이 유효하지 않을 때 제출하면 어떻게 되는가? → 이메일 형식 오류 메시지 표시
- 네트워크 오류 등 서버 오류가 발생하면 어떻게 되는가? → 일반 오류 메시지를 표시하고 폼은 유지
- 모바일 화면(320px~)에서 2-column 비밀번호 필드가 1-column으로 전환되는가? → 반응형 레이아웃 확인
- 이미 가입 완료(인증 완료)된 이메일로 회원가입을 시도하면? → OTP 화면 대신 "이미 가입된 이메일" 안내 메시지 + 로그인 이동 버튼 표시
- 가입은 했으나 이메일 인증을 완료하지 않은 이메일로 다시 시도하면? → OTP 인증 단계로 전환하여 인증 완료 유도
- 로그인된 사용자가 직접 `/signup` URL을 입력하면? → `/dashboard`로 자동 리다이렉트
- 로그인된 사용자가 직접 `/signin` URL을 입력하면? → `/dashboard`로 자동 리다이렉트

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 페이지는 로그인 페이지(`/signin`)와 동일한 헤더 구조(로고 이미지, 서비스명, 버튼)를 재사용해야 한다.
- **FR-001a**: 헤더의 로고 및 서비스명("ELUO AI SKILL HUB") 영역을 클릭하면 홈 페이지(`/`)로 이동해야 한다.
- **FR-002**: 페이지는 로그인 페이지에서 사용하는 애니메이션 배경 컴포넌트(`BackgroundBeamsWithCollision`)를 동일하게 적용해야 한다.
- **FR-003**: 회원가입 폼은 이름, 이메일, 비밀번호, 비밀번호 확인 총 4개의 입력 필드를 포함해야 한다.
- **FR-004**: 비밀번호와 비밀번호 확인 입력 필드는 모바일에서 1열, 중간 이상 화면에서 2열 그리드로 배치되어야 한다.
- **FR-005**: 폼 제출 시 비밀번호와 비밀번호 확인이 일치하지 않으면 오류 메시지를 표시하고 서버 요청을 중단해야 한다.
- **FR-006**: 폼 제출 시 이미 가입 완료(이메일 인증 완료)된 이메일이면 회원가입 폼을 대체하는 별도 글래스카드 뷰를 표시해야 한다. 카드에는 아이콘, "이미 가입된 이메일입니다" 제목, 안내 메시지, 로그인 페이지(`/signin`)로 이동하는 버튼, 그리고 "다른 이메일로 가입하기" 링크(폼으로 복귀)를 포함해야 한다.
- **FR-006a**: 폼 제출 시 이메일이 등록되어 있으나 이메일 인증이 미완료된 경우, 기존 OTP 인증 단계를 표시하여 인증을 완료할 수 있도록 해야 한다.
- **FR-007**: 폼 카드는 `signup.html`의 글래스카드 스타일(반투명 배경, 블러 효과, 라운드 코너, 최대 너비 480px)을 따라야 한다.
- **FR-008**: 카드 상단에 아이콘, 제목("회원가입"), 부제목("ELUO AI 플랫폼의 새로운 여정을 시작하세요")을 표시해야 한다.
- **FR-009**: 폼 하단에 "이미 계정이 있으신가요? 로그인" 링크가 표시되며, 클릭 시 `/signin` 페이지로 이동해야 한다.
- **FR-010**: 제출 중에는 버튼이 비활성화 상태로 전환되어 중복 제출을 방지해야 한다.
- **FR-011**: 이메일 OTP 인증 단계 카드도 회원가입 폼과 동일한 글래스카드 스타일(반투명 배경, 블러 효과, 라운드 코너, 최대 너비 480px)을 적용해야 한다.
- **FR-012**: 이미 로그인한 사용자가 `/signup` 페이지에 접근하면 `proxy.ts` 미들웨어가 `/dashboard`로 리다이렉트해야 한다.
- **FR-013**: 이미 로그인한 사용자가 `/signin` 페이지에 접근하면 `proxy.ts` 미들웨어가 `/dashboard`로 리다이렉트해야 한다.

### Key Entities

- **User (사용자)**: 이름, 이메일, 비밀번호를 통해 계정을 생성하는 신규 방문자
- **SignupForm**: 회원가입 입력 폼 (이름, 이메일, 비밀번호, 비밀번호 확인, 제출 버튼, 오류 메시지, 로그인 링크 포함)

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자가 유효한 정보를 입력하고 회원가입을 완료하는 데 2분 이내 소요된다.
- **SC-002**: 로그인 페이지와 회원가입 페이지의 헤더 및 배경이 시각적으로 동일하게 표시된다 (컴포넌트 재사용으로 확인 가능).
- **SC-003**: 폼 유효성 검사(비밀번호 불일치, 빈 필드, 잘못된 이메일 형식)가 제출 즉시 사용자에게 피드백을 제공한다.
- **SC-004**: 모바일(320px), 태블릿(768px), 데스크탑(1280px) 해상도에서 레이아웃이 정상 표시된다.

## Assumptions

- 회원가입 후 이동 경로는 기존 `005-signup-page` 구현과 동일하게 유지된다 (대시보드 또는 이메일 인증 안내 화면).
- 헤더는 별도의 공유 컴포넌트로 추출하지 않고 로그인 페이지와 동일한 JSX를 회원가입 페이지에 적용한다.
- 브랜드 컬러 토큰(`brand-navy`, `brand-yellow`, `brand-light`)은 프로젝트 Tailwind 설정에 이미 정의되어 있다.
- 비밀번호 강도 표시(strength indicator)는 이번 스코프에 포함되지 않는다.

## Clarifications

### Session 2026-03-04

- Q: OTP 인증 단계 카드에도 동일한 글래스카드 스타일을 적용해야 하는가? → A: 예, 회원가입 폼과 동일한 글래스카드 스타일을 OTP 인증 단계에도 적용한다 (FR-011 추가).
- Q: 미들웨어 파일 컨벤션은? → A: Next.js 16.x `proxy.ts` 컨벤션 사용 (`middleware.ts` 불필요).
- Q: 이미 가입된 이메일로 회원가입 시 어떤 화면을 보여주어야 하는가? → A: 인증 완료된 이메일은 OTP 화면 대신 "이미 가입된 이메일" 안내 + 로그인 이동 버튼을 제공한다. 미인증 이메일은 기존 OTP 인증 단계를 유지하여 인증을 완료하도록 유도한다.
- Q: 미인증 계정 처리 방식은? → A: OTP 미완료 = 회원가입 미완료이므로, 미인증 이메일은 OTP 인증 단계를 표시하여 인증 완료를 유도한다 (기존 동작 유지).
- Q: 중복 이메일 안내 UI 형태는? → A: OTP 단계처럼 폼을 대체하는 별도 글래스카드 뷰로 표시. 아이콘 + 제목 + 안내 메시지 + 로그인 이동 버튼 + "다른 이메일로 가입하기" 링크 포함.
- Q: "다른 이메일로 가입하기" 클릭 시 폼 데이터 유지 여부는? → A: 빈 폼으로 초기화한다.
