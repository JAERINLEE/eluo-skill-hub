# Feature Specification: 스킬 팝업 렌더링 최적화

**Feature Branch**: `018-skill-popup-optimization`
**Created**: 2026-03-04
**Status**: Draft
**Input**: User description: "스킬 팝업 렌더링 최적화 — 스킬을 클릭하면 스킬에 대한 정보와 마크다운을 보여주는데까지 너무 오래걸려. 이유가 뭔지 분석하고 수정해줘."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 스킬 팝업 즉시 응답 (Priority: P1)

사용자가 대시보드에서 스킬 카드를 클릭하면, 스킬 정보와 마크다운 콘텐츠가 체감상 즉시 표시되어야 한다. 현재는 데이터 조회가 순차적(waterfall)으로 실행되어 팝업이 열리기까지 불필요한 대기 시간이 발생하고 있다. 데이터 조회를 병렬화하여 팝업 로딩 시간을 단축한다.

**Why this priority**: 팝업 로딩 지연의 가장 큰 원인이 순차적 데이터베이스 쿼리이며, 이를 병렬화하는 것만으로 체감 속도가 크게 개선된다.

**Independent Test**: 스킬 카드 클릭 후 팝업에 스킬 정보(제목, 설명, 카테고리, 작성자, 평점)가 표시되기까지의 시간을 측정하여 개선 전후를 비교할 수 있다.

**Acceptance Scenarios**:

1. **Given** 대시보드에 스킬 카드가 표시된 상태, **When** 사용자가 스킬 카드를 클릭하면, **Then** 스킬 상세 정보(제목, 설명, 카테고리, 작성자, 평점)가 이전 대비 눈에 띄게 빠르게 표시된다.
2. **Given** 스킬에 피드백 데이터가 존재하는 상태, **When** 팝업이 열리면, **Then** 피드백 목록도 스킬 정보와 함께 빠르게 로딩된다.
3. **Given** 스킬에 작성자 프로필이 연결된 상태, **When** 팝업이 열리면, **Then** 작성자 이름이 별도의 추가 대기 없이 함께 표시된다.

---

### User Story 2 - 마크다운 콘텐츠 빠른 렌더링 (Priority: P2)

스킬 팝업 내 마크다운 콘텐츠(스킬 상세 설명)가 빠르게 렌더링되어야 한다. 현재는 마크다운 관련 라이브러리가 팝업을 열 때 한꺼번에 로드되어 초기 렌더링을 지연시키고 있다. 마크다운 렌더링에 필요한 리소스를 효율적으로 로드하여 콘텐츠 표시 속도를 개선한다.

**Why this priority**: 데이터 조회 병렬화 다음으로 큰 병목이 마크다운 라이브러리 번들 로딩이며, 이를 최적화하면 콘텐츠 표시까지의 시간이 추가로 단축된다.

**Independent Test**: 마크다운 콘텐츠가 포함된 스킬의 팝업을 열었을 때, 마크다운이 렌더링되기까지의 시간을 측정하여 개선 전후를 비교할 수 있다.

**Acceptance Scenarios**:

1. **Given** 스킬에 마크다운 콘텐츠가 있는 상태, **When** 팝업이 열리면, **Then** 마크다운 콘텐츠가 이전보다 빠르게 렌더링되어 표시된다.
2. **Given** 마크다운에 코드 블록이 포함된 상태, **When** 팝업이 열리면, **Then** 코드 하이라이팅이 적용된 상태로 표시된다.
3. **Given** 사용자가 첫 번째 스킬 팝업을 이미 열었다가 닫은 상태, **When** 두 번째 스킬 팝업을 열면, **Then** 마크다운 렌더링이 첫 번째보다 더 빠르게 완료된다 (라이브러리가 이미 로드되어 있으므로).

---

### User Story 3 - 피드백 목록 효율적 로딩 (Priority: P3)

스킬에 대한 피드백(리뷰)이 많을 경우에도 팝업 로딩 시간이 과도하게 증가하지 않아야 한다. 현재는 모든 피드백을 한 번에 조회하고 있어 피드백이 많은 스킬일수록 로딩이 느려진다. 초기 로딩 시 표시할 피드백 수를 제한하여 일관된 로딩 속도를 보장한다.

**Why this priority**: 피드백이 많은 스킬에서만 발생하는 문제이지만, 서비스가 성장할수록 영향이 커지므로 함께 최적화한다.

**Independent Test**: 피드백이 50개 이상인 스킬의 팝업을 열었을 때 로딩 시간이 피드백이 적은 스킬과 비슷한 수준인지 확인할 수 있다.

**Acceptance Scenarios**:

1. **Given** 스킬에 피드백이 다수 존재하는 상태, **When** 팝업이 열리면, **Then** 최근 피드백이 제한된 수만큼 먼저 표시되고, 나머지는 필요 시 추가 로딩할 수 있다.
2. **Given** 스킬에 피드백이 0건인 상태, **When** 팝업이 열리면, **Then** 빈 상태 메시지가 즉시 표시된다.

---

### Edge Cases

- 네트워크가 느린 환경에서 팝업을 열 때 사용자에게 로딩 상태가 명확히 표시되는가?
- 스킬 데이터 조회 중 일부 쿼리가 실패해도 나머지 정보는 정상 표시되는가?
- 마크다운 콘텐츠가 매우 긴 경우(10,000자 이상) 렌더링 성능이 허용 범위 내인가?
- 팝업을 빠르게 열었다 닫았다를 반복할 때 메모리 누수나 중복 요청이 발생하지 않는가?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 스킬 상세 조회 시 독립적인 데이터(스킬 메타데이터, 템플릿, 피드백 통계)를 병렬로 조회해야 한다.
- **FR-002**: 시스템은 피드백과 관련 답글 조회 시에도 가능한 범위에서 병렬 조회를 수행해야 한다.
- **FR-003**: 시스템은 피드백 초기 조회 시 최근 20건으로 제한하여 조회해야 한다.
- **FR-004**: 마크다운 렌더링에 필요한 리소스는 팝업이 열릴 때 효율적으로 로드되어야 하며, 초기 페이지 로딩에 영향을 주지 않아야 한다.
- **FR-005**: 마크다운 렌더링 플러그인(구문 하이라이팅 등)은 해당 콘텐츠가 존재할 때만 처리되어야 한다.
- **FR-006**: 팝업을 빠르게 열고 닫을 때 불필요한 중복 요청이나 메모리 누수가 발생하지 않아야 한다.
- **FR-007**: 기존 스킬 팝업의 모든 기능(스킬 정보 표시, 마크다운 렌더링, 피드백 표시, 템플릿 다운로드)이 최적화 후에도 동일하게 동작해야 한다.

### Key Entities

- **Skill**: 스킬 메타데이터(제목, 설명, 카테고리, 작성자), 마크다운 콘텐츠, 템플릿 파일, 피드백 통계
- **Feedback**: 사용자 리뷰(평점, 코멘트), 답글, 작성자 프로필
- **Template**: 스킬에 첨부된 다운로드 가능한 파일

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 스킬 카드 클릭 후 팝업에 스킬 정보가 표시되기까지 걸리는 시간이 최적화 전 대비 40% 이상 단축된다.
- **SC-002**: 피드백이 50건 이상인 스킬과 5건 미만인 스킬의 팝업 로딩 시간 차이가 200ms 이내이다.
- **SC-003**: 팝업을 두 번째 이후로 열 때 마크다운 렌더링 시간이 첫 번째 대비 50% 이상 단축된다.
- **SC-004**: 최적화 후 기존 스킬 팝업 기능의 100%가 정상 동작한다 (회귀 없음).

## Assumptions

- 현재 팝업 로딩 지연의 주요 원인은 순차적 데이터베이스 쿼리(waterfall pattern)와 마크다운 라이브러리 번들 로딩이다.
- 피드백 초기 조회 제한 건수는 20건으로 가정한다. 이후 "더 보기"로 추가 로딩이 가능하다.
- 마크다운 라이브러리의 지연 로딩(lazy loading)이 기술적으로 가능하다.
- 데이터베이스 쿼리 병렬화 시 Supabase 연결 풀이 동시 요청을 처리할 수 있다.

## Scope & Boundaries

### In Scope
- 스킬 상세 조회 데이터베이스 쿼리 병렬화
- 피드백 조회 쿼리 최적화 및 초기 로딩 제한
- 마크다운 렌더링 리소스의 효율적 로딩
- 마크다운 플러그인 조건부 적용

### Out of Scope
- 스킬 팝업 UI/UX 디자인 변경
- 새로운 기능 추가
- 서버 인프라 변경 (CDN, 캐시 서버 등)
- 대시보드 전체 성능 최적화 (팝업 외 영역)
