## Clarifications

### Session 2026-03-04

## 현재 상태 분석 결과

### 발견된 문제점

1. **스킬 CRUD 파일 작업 순차 실행**: 스킬 생성·수정·삭제 시 마크다운 파일 업로드, 템플릿 파일 업로드, 피드백 삭제 등의 작업이 순차적으로 실행되어 관리자 대기 시간이 누적됨
2. **서버 액션 데이터 무효화 누락**: 스킬 생성·수정 후 관련 페이지 무효화가 호출되지 않아 관리 목록에 변경사항이 즉시 반영되지 않는 경우 발생
3. **멤버 역할 변경 후 무효화 누락**: 멤버 역할 변경 서버 액션에서 관련 페이지 무효화가 호출되지 않아 변경 후 최신 상태가 표시되지 않음
4. **어드민 마크다운 프리뷰 스타일 불일치**: 어드민 스킬 생성·수정 시 마크다운 프리뷰가 GitHub 라이트 테마를 사용하고 있어, 대시보드 스킬 팝업의 노션 스타일 렌더링과 시각적으로 다름
5. **서버 액션 `verifyAdmin()` 중복 호출**: 스킬 CRUD 서버 액션(`createSkill`, `updateSkill`, `getSkillById`, `getCategories`)이 각각 독립적으로 관리자 인증을 수행하여, 수정 팝업 오픈 시 불필요한 인증 쿼리가 중복 발생

### 분석 결론

어드민 페이지의 데이터 조회(대시보드, 스킬 목록, 멤버 목록)는 이미 병렬화가 잘 적용되어 있다. 주요 개선 대상은 **CRUD 작업 시 파일 처리 병렬화**, **서버 액션의 데이터 무효화 누락 수정**, **마크다운 프리뷰 스타일 통일**, **`verifyAdmin()` 중복 호출 최적화**이다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 스킬 CRUD 작업 속도 개선 (Priority: P1)

관리자가 스킬을 생성·수정·삭제할 때, 마크다운 파일과 템플릿 파일에 대한 처리(업로드, 삭제)가 가능한 한 동시에 실행되어 작업 완료까지의 대기 시간이 줄어든다. 현재 파일 업로드가 하나씩 순차적으로 진행되어 파일 수가 많을수록 대기 시간이 길어지는 문제를 개선한다. 또한 서버 액션의 `verifyAdmin()` 중복 호출을 최적화하여 수정 팝업 오픈 시 불필요한 인증 쿼리를 줄인다.

**Why this priority**: 관리자가 가장 자주 수행하는 스킬 관리 작업의 체감 속도에 직접적으로 영향을 미침. 특히 템플릿 파일이 여러 개인 스킬의 경우 대기 시간이 크게 체감됨

**Independent Test**: 템플릿 파일 3개 이상을 포함한 스킬을 생성·수정·삭제하면서 작업 완료 시간을 측정하고, 최적화 전과 비교

**Acceptance Scenarios**:

1. **Given** 관리자가 마크다운 파일 1개와 템플릿 파일 3개를 포함한 새 스킬을 생성할 때, **When** 스킬 저장을 실행하면, **Then** 파일 업로드가 동시에 처리되어 이전 대비 빠르게 완료된다
2. **Given** 관리자가 기존 스킬의 마크다운 파일을 교체하고 템플릿 파일 2개를 추가할 때, **When** 스킬 수정을 실행하면, **Then** 파일 삭제와 업로드가 동시에 처리되어 이전 대비 빠르게 완료된다
3. **Given** 관리자가 템플릿 파일이 여러 개인 스킬을 삭제할 때, **When** 삭제를 실행하면, **Then** 관련 파일과 데이터 정리가 동시에 처리되어 이전 대비 빠르게 완료된다
4. **Given** 파일 업로드 중 일부가 실패할 때, **When** 작업이 완료되면, **Then** 실패한 파일에 대한 오류 메시지가 표시되고, 성공한 파일은 정상적으로 저장된다
5. **Given** 관리자가 수정 팝업을 열 때, **When** 스킬 데이터와 카테고리를 로딩하면, **Then** 관리자 인증 확인이 중복 없이 최소 횟수로 수행된다

---

### User Story 2 - 데이터 변경 후 관리 화면 즉시 반영 (Priority: P2)

관리자가 스킬 생성, 스킬 수정, 멤버 역할 변경 등의 데이터를 변경하는 행위를 수행한 후, 관련 관리 페이지에 돌아왔을 때 변경사항이 즉시 반영되어 있다. 현재 일부 서버 액션에서 데이터 무효화가 누락되어 이전 상태가 표시되는 문제를 해결한다.

**Why this priority**: 관리자가 변경한 내용이 목록에 즉시 반영되지 않으면 작업이 실패했다고 오인하거나 중복 작업을 수행할 수 있음

**Independent Test**: 스킬 생성 후 스킬 목록으로 돌아가서 새 스킬이 표시되는지 확인. 멤버 역할 변경 후 목록에 변경된 역할이 표시되는지 확인.

**Acceptance Scenarios**:

1. **Given** 관리자가 새 스킬을 생성할 때, **When** 저장 완료 후 스킬 목록 페이지로 이동하면, **Then** 새로 생성한 스킬이 목록에 즉시 표시된다
2. **Given** 관리자가 기존 스킬을 수정할 때, **When** 수정 완료 후 스킬 목록 페이지로 이동하면, **Then** 수정된 내용이 즉시 반영되어 표시된다
3. **Given** 관리자가 멤버의 역할을 변경할 때, **When** 변경이 완료되면, **Then** 멤버 목록에서 변경된 역할이 즉시 표시된다
4. **Given** 관리자가 스킬을 삭제한 후 대시보드로 이동할 때, **When** 대시보드가 표시되면, **Then** 대시보드의 스킬 통계가 최신 상태로 반영되어 있다

---

### User Story 3 - 어드민 마크다운 프리뷰 노션 스타일 통일 (Priority: P3)

관리자가 스킬 생성·수정 폼에서 마크다운 파일을 업로드하면, 프리뷰 영역에 대시보드 스킬 상세 팝업과 동일한 노션 스타일 렌더링이 적용된다. 현재 어드민의 마크다운 프리뷰는 GitHub 라이트 테마를 사용하고 있어 대시보드의 노션 스타일과 시각적으로 불일치하는 문제를 해결한다.

**Why this priority**: 관리자가 작성한 마크다운 콘텐츠가 사용자에게 어떻게 보일지 정확히 미리 확인할 수 있어야 편집 효율이 높아짐. WYSIWYG 경험 제공

**Independent Test**: 어드민 스킬 생성/수정 폼에서 마크다운 파일을 업로드한 뒤 프리뷰가 대시보드 스킬 팝업과 동일한 스타일로 렌더링되는지 시각적으로 비교

**Acceptance Scenarios**:

1. **Given** 관리자가 스킬 생성 폼에서 마크다운 파일을 업로드할 때, **When** 프리뷰가 표시되면, **Then** 대시보드 스킬 상세 팝업과 동일한 노션 스타일(코드 블록 다크 테마, 테이블 노션 스타일, 인라인 코드 레드 텍스트 등)이 적용된다
2. **Given** 관리자가 스킬 수정 팝업에서 기존 마크다운을 확인할 때, **When** 프리뷰가 표시되면, **Then** 사용자가 대시보드에서 보는 것과 동일한 렌더링 결과를 확인할 수 있다
3. **Given** 마크다운에 frontmatter 메타데이터가 포함된 경우, **When** 어드민 프리뷰에서 표시되면, **Then** 대시보드와 동일하게 frontmatter 카드가 렌더링된다

---

### Edge Cases

- 스킬 생성 중 파일 업로드가 일부만 성공하면 어떻게 되는가? → 성공한 파일만 저장하고 실패 항목에 대해 오류를 반환
- 네트워크 지연 환경에서 다수의 파일을 동시 업로드할 때 타임아웃이 발생하면? → 개별 파일별로 오류를 처리하여 나머지 파일은 정상 업로드
- 스킬 수정 중 기존 파일 삭제는 성공했으나 새 파일 업로드가 실패하면? → 오류 메시지를 반환하여 관리자가 재시도할 수 있도록 안내
- 동시에 두 관리자가 같은 스킬을 수정하면? → 마지막 저장이 우선되며 기존 동작을 유지
- 마크다운 프리뷰 스타일 변경 시 기존 어드민 UI 레이아웃에 영향을 주는가? → 노션 스타일이 `.notion-markdown` 클래스로 스코핑되어 다른 UI에 영향 없음

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 스킬 생성 시 마크다운 파일 업로드와 템플릿 파일 업로드가 동시에 실행되어야 한다
- **FR-002**: 스킬 수정 시 기존 파일 삭제와 새 파일 업로드가 가능한 한 동시에 실행되어야 한다
- **FR-003**: 스킬 삭제 시 관련 피드백 데이터 삭제와 파일 삭제가 동시에 실행되어야 한다
- **FR-004**: 파일 처리 중 개별 작업이 실패해도 나머지 작업은 정상적으로 완료되어야 한다
- **FR-005**: 스킬 생성 서버 액션 완료 후 관련 관리 페이지의 데이터가 무효화되어야 한다
- **FR-006**: 스킬 수정 서버 액션 완료 후 관련 관리 페이지의 데이터가 무효화되어야 한다
- **FR-007**: 멤버 역할 변경 서버 액션 완료 후 관련 관리 페이지의 데이터가 무효화되어야 한다
- **FR-008**: 기존의 스킬 검색, 상태 필터, 카테고리 필터, 페이지네이션 기능이 최적화 후에도 동일하게 동작해야 한다
- **FR-009**: 기존의 멤버 검색, 역할 변경, 자기 자신 역할 변경 차단 기능이 동일하게 동작해야 한다
- **FR-010**: 어드민 스킬 생성·수정 폼의 마크다운 프리뷰가 대시보드 스킬 상세 팝업과 동일한 노션 스타일로 렌더링되어야 한다
- **FR-011**: 마크다운 프리뷰의 노션 스타일은 어드민 UI의 다른 요소에 영향을 주지 않도록 스코핑되어야 한다
- **FR-012**: 수정 팝업 오픈 시 관리자 인증 확인이 중복 없이 최소 횟수로 수행되어야 한다

### Key Entities

- **스킬**: 관리자가 등록하는 자동화 스킬 (마크다운 설명서 + 템플릿 파일 포함)
- **스킬 템플릿 파일**: 스킬에 첨부되는 다운로드 가능한 파일 (0~N개)
- **멤버**: 시스템에 등록된 사용자 (역할: admin, viewer 등)
- **피드백**: 스킬에 대한 사용자 평가 (삭제 시 스킬과 함께 삭제됨)

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 템플릿 파일 3개 이상을 포함한 스킬 생성·수정 시 작업 완료 시간이 최적화 전 대비 30% 이상 단축된다
- **SC-002**: 스킬 생성·수정·삭제 후 100%의 경우 관련 관리 페이지에서 최신 데이터가 즉시 표시된다
- **SC-003**: 멤버 역할 변경 후 100%의 경우 관련 페이지에서 최신 역할이 즉시 표시된다
- **SC-004**: 모든 기존 관리 기능(스킬 CRUD, 멤버 관리, 피드백 조회, 대시보드)이 최적화 전과 동일하게 동작한다
- **SC-005**: 어드민 마크다운 프리뷰가 대시보드 스킬 팝업과 시각적으로 동일한 렌더링 결과를 보여준다

## Assumptions

- 어드민 페이지의 데이터 조회(대시보드, 스킬 목록, 멤버 목록)는 이미 병렬화되어 있으므로, 이번 피처에서는 조회 병렬화가 아닌 CRUD 작업 최적화에 집중한다
- 파일 스토리지는 동시 업로드/삭제 요청을 지원한다고 가정한다
- 대시보드의 노션 스타일 마크다운 컴포넌트(`NotionStyleMarkdown`)와 관련 CSS(`notion-markdown.css`)는 이미 구현되어 있으며 재사용 가능하다

## Scope Boundaries

### In Scope

- 스킬 생성·수정·삭제 서버 액션의 파일 처리 병렬화
- 스킬 CRUD 서버 액션의 데이터 무효화 누락 수정
- 멤버 역할 변경 서버 액션의 데이터 무효화 누락 수정
- 어드민 마크다운 프리뷰를 대시보드와 동일한 노션 스타일로 변경
- 수정 팝업의 `verifyAdmin()` 중복 호출 최적화

### Out of Scope

- 어드민 데이터 조회 병렬화 (이미 완료됨)
- 카테고리/역할 데이터의 캐싱 레이어 추가 (현 규모에 불필요)
- 어드민 페이지의 클라이언트 사이드 렌더링 전환 (서버 컴포넌트 패턴 유지)
- DB 인덱스 최적화 (DBA 영역)
